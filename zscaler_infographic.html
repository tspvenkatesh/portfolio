ZCC Overview

Lightweight endpoint agent installed on user devices (Windows, macOS, iOS, Android). Acts as the traffic interception and steering engine — the brain that decides which traffic goes to ZIA vs ZPA vs bypassed.

// ZCC Traffic Steering Methods
METHOD 01
Tunnel Mode
All traffic from the device is encapsulated in an IPSec/GRE tunnel to the nearest Zscaler Enforcement Node (ZEN). Full inspection applies regardless of destination port.
ZIA default path
METHOD 02
PAC File
ZCC auto-provisions a Proxy Auto-Config file. Browser/OS-level proxy redirection to ZIA cloud proxy. Works well for HTTP/HTTPS-only traffic in legacy environments.
HTTP/HTTPS only
METHOD 03
Z-App Tunnel
ZPA-specific lightweight per-app tunnel. Traffic to private application segments is micro-tunneled to ZPA Broker Cloud, connecting via App Connector on-prem or in cloud.
Private app access
METHOD 04
Split Tunneling
Selective steering — only matched destinations (FQDN, IP range, app category) go to ZIA/ZPA. Remaining traffic exits locally. Reduces latency for trusted internal traffic.
Performance tuning
METHOD 05
Bypass Rules
Explicit bypass list for traffic that must not transit Zscaler — e.g., local printers, DHCP, internal monitoring agents, or legacy protocols that cannot tolerate proxy interruption.
Operational exclusions

SIPA — Source IP Anchoring
// WHAT IS SIPA

SIPA (Source IP Anchoring) is a ZPA feature that allows a user's traffic to egress from a fixed, predictable source IP address — typically a corporate static IP — even when the user is remote. Traffic is tunneled via ZPA to an App Connector deployed on-prem or in a cloud region, and exits from that connector's IP. This allows destination systems (SaaS, partner portals, legacy apps) that enforce IP allowlisting to continue functioning for remote users — without needing a traditional VPN.

Why SIPA Is Needed
IP Allowlisting: SaaS apps, partner portals, and banking systems that restrict access to known corporate IPs
Legacy app constraints: On-prem or hosted apps that cannot be converted to identity-based auth
Compliance requirements: Regulatory mandates that traffic must originate from specific geographies or IP ranges
VPN elimination: Remove split-tunnel VPN while preserving the static source IP behavior remote users depend on
Audit trails: Systems that log source IPs for audit — SIPA ensures consistent identifiable egress
⚙️
How SIPA Works
ZCC on the endpoint matches traffic destined for SIPA-configured domains/IPs
Traffic is tunneled over ZPA micro-tunnel to a designated App Connector Group
App Connector forwards traffic to the internet/SaaS destination using its own static egress IP
Return traffic flows back through the same App Connector → ZPA Broker → ZCC to the user
No inbound ports required on the App Connector host — all connections are outbound-initiated
🏗
SIPA Architecture Components
App Connector: Deployed in DC or cloud VPC with a static public IP (NAT gateway or Elastic IP on AWS/Azure)
App Segment (SIPA type): Defined with destination FQDN/IP of the target SaaS or application
Server Group: Linked to the App Connector Group responsible for egress
Access Policy: ZPA access policy scoped to user group that needs anchored IP access
No App Connector needed at app side — connector acts as an egress proxy, not a gateway to internal app
⚠️
SIPA Considerations & Limits
Not for all traffic: SIPA is selective — only configured App Segments use anchoring; rest uses ZIA
Connector sizing: High-volume SIPA flows (video, large file transfers) require adequately sized connector VMs
Geo-anchoring: If compliance requires country-specific egress, deploy connectors in the right region/cloud
HA / redundancy: Deploy multiple App Connectors per group for failover — ZPA load-balances automatically
SIPA ≠ ZIA bypass: If ZIA SSL inspection is also enabled, traffic may still pass through ZEN before the connector
